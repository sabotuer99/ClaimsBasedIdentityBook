Claims Aware Web Application in a Web Farm

This sample demonstrates how to run a relying party in a web farm scenaro. The relying party enumerates issued claims in an ASP.NET web application. The sample includes a Security Token Service (STS) and shows how to obtain a federated token from this STS using the WS-Federation Passive protocol. In addition to this STS, this sample includes a web application that demonstrates a mechanism for obtaining the federated token. This application serves as a “relying party” in the overall sample and has a trust relationship with the STS.

  PassiveRedirectBasedClaimsAwareWebAppInWebFarm

This web application illustrates the use of the passive-redirect feature of the WSFederationAuthenticationModule object available in the Windows Identity Foundation. When a user browses this web site, the WSFederationAuthenticationModule checks to see if he or she is authenticated. If the user is not authenticated, the browser will redirect automatically to the STS web site. The cookies are protected using the relying party's service certificate, in order to enable this web application to run in a web farm. 

As mentioned below, the STS web site is configured to have Windows authentication enabled. Once the client authenticates to the STS via Windows authentication, the STS will issue a SAML token containing claims based on the authenticating user. This token is sent back to the relying party web site via the browser. Both relying party web applications in this sample will enumerate and display the issued claims on the browser.

Projects included in the sample

PassiveRedirectBasedClaimsAwareWebAppInWebFarm

TokenReplayCacheService

The TokenReplayCache feature of WIF is designed to ensure that originating tokens used to authenticate callers are only used once.  By default, when validating Saml11 or Saml2 tokens that do not contain keys, a check is made to ensure that the token has not been processed.  This helps protect against replay attacks.   This sample includes an example, using WCF, of a shared  TokenReplayCache that can be used in a web farm.  A self hosted WCF service implements a TokenReplayCache.  In a production environment, it may be advantageous to experiment with different bindings to achieve performance goals.  Comments are included in web.config of the TokenReplayCacheService.  It is important in a WebFarm scenario to have only one TokenReplayCache that is shared between the services as the replayed token may be presented to any service in the farm.  If each service has its own TokenReplayCache, then an attacker can successfully attack using replayed tokens.

Using the ASP.NET Machine Key feature to encrypt session cookies

Sometimes, a decryption key doesn’t make sense for your relying party and all your application requires is straightforward symmetric key encryption for session cookies. In this deployment, a shared secret in the form of a symmetric key is deployed in the web.config files in all of the relying party web sites in a web farm. This sample demonstrates how to do this using the ASP.NET Machine Key feature. You can read more about the details of the Machine Key feature on MSDN by reading the “HOWTO: Configure MachineKey in ASP.NET 2.0” article.

To explore this optional part of this sample, several additional steps are required. The steps prefixed with “IIS7 Only” are only possible on machines with IIS version 7. The HOWTO article above shows how to configure Machine Keys on IIS version 6.

1.       IIS7 Only.  Open the Internet Information Services (IIS) Manager administration tool. Browse to the “PassiveRedirectBasedClaimsAwareWebAppInWebFarm” virtual directory in the “Connections” pane. Open the “Machine Key” feature.

2.       IIS7 Only. In the Machine Key feature configuration, make the following changes:

a.       Encryption Method: SHA1

b.      Decryption Method: AES

c.       Validation Key: Uncheck “Automatically generate at runtime” and uncheck “Generate a unique key for each application”

d.      Decryption Key: Uncheck “Automatically generate at runtime” and uncheck “Generate a unique key for each application”

3.       IIS7 Only. On the “Actions” pane, click the “Generate Keys” link to generate a new set of keys for your application. Click “Apply” to save these keys to the web.config file for this application.

4.       In the web.config file, find the line containing adding the RsaEncryptedSessionSecurityTokenHandler to the service’s token handlers collection. Comment this line out and as per the XML comments instructions, uncomment the line below which adds the MachineKeySessionSecurityTokenHandler.

5.       Re-deploy your application in your farm since you made changes to the web.config file.

6.       Close any instance of your web browser that may be logged into this sample already and open a new browser window. This step is necessary to clear out the session cookie in this sample.

7.       Browse https://localhost/PassiveRedirectBasedClaimsAwareWebAppInWebFarm/default.aspx
