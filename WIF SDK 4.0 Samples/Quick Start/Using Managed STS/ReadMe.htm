Claims Aware Web Application Using Managed STS

The purpose of this sample is to show how claims are enumerated in an ASP.NET web application, from a token issued by a managed Security Token Service (STS) like one hosted in the cloud. Examples  of this would be the Live ID IP STS or STSs on the corporate network using the WS-Federation Passive protocol like Active Directory Federation Services 2.0.

This sample illustrates the use of the passive-redirect feature of the WS-FederationAuthenticationModule available in the Windows Identity Foundation. When the user browses to the relying party web site, the WS-FederationAuthenticationModule checks to see if the user is authenticated. If not, the user is automatically redirected to the STS web site. Once the client has authenticated to the STS, the STS issues a SAML token with claims based on the authenticated user. This token is sent via the browser back to the relying party. At this point the token is processed to enumerate and display the issued claims. This sample also shows how to run the Windows Identity Foundation Federation Utility to configure a pre-existing installation of Active Directory Federation Services 2.0 or Windows Live ID as a managed STS.

Project included

ClaimsAwareWebAppWithManagedSTS: An ASP.NET Web application that uses the passive-redirect feature of the WS-FederationAuthenticationModule available in the Windows Identity Foundation to obtain a token from identity providers such as Active Directory Federation Services 2.0 or Windows Live ID for federated authentication.

You can do this by hitting F12 when you are in http://msm.live.com and in the Microsoft Services Manager (MSM) for Windows LiveID Developer Tools window click 'Browse Mode' menu and then select Internet Explorer 7 option from the list.

2.       When prompted to login to the LiveID sign-in page, use a valid LiveID user account to log in (if you dont have an account go to http://login.live.com and click on Sign up for an account to create one).

3.       Click Accept Terms in the Microsoft Services Manager Terms of Use and Notices page.

4.       Enter a company name and a contact E-mail and click Update. A warning message will show up telling that you have to register a new site.

No sites warning message

6.       Enter a friendly name as the name for your site in Type a name for your site edit box (i.e. FederationwithLiveID).

7.       Enter a dummy DNS Value in the Type a DNS Name for your site in INT environment edit box (for example: something.com). Note that it doesnt need to match your actual DNS name because in step 11-c below we would be modifying this value to be urn value.

8.       Click the Windows Live ID radio button and skip the rest of the fields.

Register Your Site form

 

9.       Click Submit to save the changes.

10.       On the Confirmation page click Yes.

Confirmation page

 

You may notice an Error message telling the DNS name is already in use, in which case, Click the Go to Manage Your Site link and start from step-6 above.

Step-2 - Provisioning Windows LiveID as STS to a Relying Party Application

1.       Open Microsoft Visual Studio with administrator privileges.

2.       Open the solution file for this sample.

3.       On the Solution Explorer, right-click the project and select Add STS reference menu item.

4.       On the Welcome dialog, in the Application URI enter the DNS name for the site (the one that you've entered in Task1 - for example, urn:foo:foo2); and click Next.

Configuring the relying party

 

5.       You will get a warning stating that the application is not hosted in a secure connection. Click Yes.

Security warning

 

6.       On the STS options dialog select the third radio button "Use an existing STS" option, enter https://nexus.passport-int.com/federationmetadata/2007-06/federationmetadata.xml as STS metadata location and click Next.

Configuring the endpoint metadata

7.       The next screen will be the Certificate Trust Error screen. Choose the option based on whether certificate validation is required or not. NOTE: For production environments this is not recommended but for purposes of this sample validation can be disabled. In the case of using certificate validation, all the certificates that have failed validation will be saved to the desktop by the Federation Utility tool and they need to be imported into the Trusted People certificate store of the user account running the service.

 

8.       Select No encryption and click Next.

Configuring tokens encryption

 

9.       In the Claims Offered dialog review the claims list and then click Next.

10.   In the Summary dialog review the changes to be made by FedUtil and click Finish.

FedUtil would update the web.config file of the application with the trust information for LiveID STS.

Step-3: Updates to web.config file

 

1.       Open the Web.config file of the application.

2.       Register the CustomIssuerTokenResolver. To do this, add the following line under the microsoft.ServiceModel\service element.

<microsoft.identityModel>

<service>

...

<issuerTokenResolver type="CustomIssuerTokenResolver"/>

...

</service>

</microsoft.identityModel>

 

3.       Configure the securityTokenHandlers element to require the nameidentifier claim. To do this, add the following configuration to the microsoft.ServiceModel\service element.

<microsoft.identityModel>

<service>

...

<issuerTokenResolver type="CustomIssuerTokenResolver"/>

<securityTokenHandlers>

    <remove type="Microsoft.IdentityModel.Tokens.Saml11.Saml11SecurityTokenHandler, Microsoft.IdentityModel, Version=3.5.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35" />

    <add type="Microsoft.IdentityModel.Tokens.Saml11.Saml11SecurityTokenHandler, Microsoft.IdentityModel, Version=3.5.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35">

       <samlSecurityTokenRequirement>

            <nameClaimType value="http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier"/>

       </samlSecurityTokenRequirement>

</add>

</securityTokenHandlers>

...

</service>

</microsoft.identityModel>

4.       Save the changes to web.config file

 

Step-4 Accessing the application

1.       If you dont have a Live ID account from before then browse to the https://login.live-int.com/login.srf and create a Live ID user account and then sign-out.

2.       Browse to https://localhost/ClaimsAwareWebAppWithManagedSTS/default.aspx; you will be redirected to a LiveID login page; login with the Live ID user account you created above.

Notice that you will be redirected to your relying party application and the claims issued by LiveID are displayed in the page. The output of claims display page would look like below.

Use Active Directory Federation Services 2.0 as an IP-STS

Before proceeding with the below steps make sure to run Setup.bat, which sets up the virtual directory for the application in IIS.

Step-1 - Provisioning Active Directory Federation Services 2.0 as STS to a Relying Party Application

1.       Open Microsoft Visual Studio with administrator privileges.

2.       Open the solution file for this sample.

3.       On the Solution Explorer, right-click the project and select Add STS reference menu item.

4.       On the Welcome dialog, in the Application URI, enter https://localhost/ClaimsAwareWebAppWithManagedSTS/ and click Next.

Note that if you have ran through LiveID STS exercise before this exercise you would notice the fields are pre-populated in the FedUtil dialogs. However, make sure to verify the values pre-populated are appropriate for this exercise and when needed make the necessary modifications to match the values described in this exercise.

Configuring the relying party

 

5.       On the STS options dialog select the third radio button "Use an existing STS" option, enter the location of federation metadata document for the STS. For this exercise we would use a Active Directory Federation Services 2.0 STS that is deployed online for evaluation purposes and enter https://ip-sts-01.federatedidentity.net/federationmetadata/2007-06/federationmetadata.xml for STS metadata location and click Next. Note that this online Active Directory Federation Services 2.0 STS is only for evaluation purposes and intended for this sample only, it should not be used for production applications.

In the case where you have installed Active Directory Federation Services 2.0 STS in your network then the federation metadata document is available at https://<host name of the STS>/federationmetadata/2007-06/federationmetadata.xml. For example if it is installed with the host name fabrikam.com then federation metadata document would be available at https://fabrikam.com/federationmetadata/2007-06/federationmetadata.xml.

Configuring the endpoint metadata

6.       The next screen will be the Certificate Trust Error screen. Choose the option based on whether certificate validation is required or not. NOTE: For production environments this is not recommended but for purposes of this sample validation can be disabled. In the case of using certificate validation, all the certificates that have failed validation will be saved to the desktop by the Federation Utility tool and they need to be imported into the Trusted People certificate store of the user account running the service.

7.       In the Security token encryption dialog, select No encryption and click Next.

Configuring token encryption

8.       In the Offered claims dialog review the claims offered list and then click Next.

Claims offered list

If you would want to modify the list of claims requested by the application, open the web.config file and edit the claims requested list under <claimTypeRequired> element under <applicationService> element in <microsoft.identityModel> section.

9.       In the Summary dialog review the changes to be made by FedUtil tool and click Finish.

FedUtil would update the web.config file of the application with the trust information for the STS.

Step-2: Updates to web.config file if you ran LiveID exercise before

This step is required only when you run the LiveID exercise before using Active Directory Federation Services 2.0 STS.

1.       Open the Web.config file of the application.

2.       Delete the <issuerTokenResolver> element and <securityTokenHandlers> element.

<microsoft.identityModel>

<service>

...

    <issuerTokenResolver type="CustomIssuerTokenResolver"/>

    <securityTokenHandlers>

      <remove type="Microsoft.IdentityModel.Tokens.Saml11.Saml11SecurityTokenHandler, Microsoft.IdentityModel, Version=3.5.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35" />

      <add type="Microsoft.IdentityModel.Tokens.Saml11.Saml11SecurityTokenHandler, Microsoft.IdentityModel, Version=3.5.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35">

        <samlSecurityTokenRequirement>

          <nameClaimType value="http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier"/>

        </samlSecurityTokenRequirement>

      </add>

    </securityTokenHandlers>

...

</service>

</microsoft.identityModel>

3.       Save the changes to web.config file
