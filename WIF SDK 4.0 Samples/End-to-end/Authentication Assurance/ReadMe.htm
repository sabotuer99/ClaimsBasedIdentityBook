Achieving Authentication Assurance with Claims

The purpose of this sample is to show how the claims model can be used to enforce a certain level of authentication assurance within various operations in a given application. The usage of Certificates or Smart Card for authenticating as high assurance in this sample is for illustration purposes only. Applications should analyze their requirement and choose a stronger authentication mechanism based on their business needs.

Projects included

1.       AuthAssuranceRP: Relying party that has a requirement for different authentication assurance levels for various processes (Account summary page access vs. high-value transaction page access)

2.       AuthAssuranceSTS: Security token service that provides two authentication methods to obtain a token. Based on the ‘wauth’ parameter in the incoming request, it authenticates a user using Windows Authentication or by using a Information Card backed with X.509 Certificate.

Running the sample

·         Access https://localhost/AuthAssuranceSTS/WindowsAuth/DownloadCard.aspx to download and install the Information Card. The Information Card is backed by a test X.509 Certificate “CN=bobclient”.

·         Access https://localhost/AuthAssuranceRP/Default.aspx

·         Select a resource (low value or high value) and go through the flow.

·         When you select ‘Low value’ resources, it requires you to login with Windows Authentication at the STS.

·         When you select ‘High value’ resources, it requires you to login with a Information Card at the STS. Submit the Information Card that you installed. The user will be authenticated based on the X.509 Certificate that is backing the Information Card.

·         If you logged on with Information Card, you will see 5 claims (Name, Authentication Type, Date of Birth, Postal code, Phone Number). Once you have logged in with high assurance you will have these 5 claims even when visiting a low assurance page.

·         If you logged on with Windows Authentication, you will see only 2 claims (Name and Authentication Type claim).

·         To log out of the session, click the ‘Sign Out’ button or close the browser.

Troubleshooting tips

Audience URI checks performed in the WSFederationAuthenticationModule in the RP are case-sensitive. If you have trouble signing in at the WSFederationAuthenticationModule, check if the Relying Party URI specified to the STS (usually using the ‘wtrealm’ parameter in a Passive sign-in scenario) matches the list of configured allowed Audience URIs in the SAML token handler configuration.

Note

The relying party application(s) in this sample can accept tokens issued by an Active Directory Federation Services 2.0 STS instance instead of the sample STS instance. You would need to modify the STS configuration parameters such as ‘issuer’ in the application’s configuration. Refer to the AD FS 2.0 product documentation for instructions on how to register a relying party application with an AD FS 2.0 STS instance.
