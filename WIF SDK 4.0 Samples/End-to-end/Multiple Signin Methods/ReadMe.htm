Web App with Multiple SignIn Methods – Windows Identity Foundation Sample

The purpose of this sample is to illustrate how an ASP.NET web application (relying party) can be built using Windows Identity Foundation to authenticate users through one of multiple authentication methods.

Projects included

1.       RelyingParty: ASP.NET web application that authenticates users through one of multiple authentication methods viz. (i) tokens from an STS that uses Forms Authentication (ii) tokens from an STS that uses Windows Authentication

2.       STSForms: Security token service that uses Forms Authentication and issues tokens.

4.       STSWindows: Security token service that uses Windows Authentication and issues tokens.

Steps to run the sample

1.       Run the Setup.bat batch file with administrative privileges to setup the sample. This includes creating the following virtual directories in IIS with the specified authentication method(s):

2.       Build the MultiAuth solution.

Running the sample

·         Access https://localhost/MultiAuthRP/Default.aspx

·         Select one of the authentication methods to sign-in

·         Depending on your selection, you would be redirected to a suitable STS to obtain a token.

         If you chose “Windows”, you would be authenticated using windows-integrated authentication

         If you chose “Forms”, you would need to login with “JohnDoe” as the username and “JohnDoe123” as the password.

 Note:

1.       The mode of storing user credentials for forms-authentication in web.config is used in this sample for illustrative purposes only. In a real deployment, consider using a membership provider to store and validate user credentials.

2.       The password found in the web.config is the SHA1 hash of "JohnDoe123" and was generated using FormsAuthentication.HashPasswordForStoringInConfigFile() method

3.       You may add additional users by editing the <credentials> section of the web.config file in the STSForms website or by configuring a suitable membership provider

·         On successful authentication to the STS, you will be issued a token and redirected to the relying party.

·         The relying party displays the set of claims found in the token (the claims would depend upon the STS from which you obtain the token)

Security Note

Note that the mode of storing user credentials for forms-authentication in web.config is used in this sample for illustrative purposes only. In a real deployment, consider using a membership provider to store and validate user credentials.

Note

The relying party application(s) in this sample can accept tokens issued by an Active Directory Federation Services 2.0 STS instance instead of the sample STS instance. You would need to modify the STS configuration parameters such as ‘issuer’ in the application’s configuration. Refer to the AD FS 2.0 product documentation for instructions on how to register a relying party application with an AD FS 2.0 STS instance.
